# 一树企业 记忆架构与上下文注入设计

## 一、四级材料体系

### L1 一级原材料（原子事实）

从单次对话或外部输入中提取的不可再分解的事实碎片。

| 子类 | 存储 | 示例 | 保留策略 |
|------|------|------|---------|
| memory_entries.identity | memory_entries | "张总，北京，精密制造，40人" | 永久，几乎不衰减 |
| memory_entries.preference | memory_entries | "偏好用关系解决问题" | 永久，慢衰减 |
| memory_entries.behavior | memory_entries | "每次员工犯错都亲自处理" | 永久，慢衰减 |
| memory_entries.business_fact | memory_entries | "每月10号报价审批老板亲自做" | 永久，中速衰减 |
| memory_entries.employee_fact | memory_entries | "王磊4/7私自给A客户打八折" | 永久，中速衰减 |
| memory_entries.emotional_state | memory_entries | "对数字化转型焦虑但不想承认" | 永久，快衰减 |
| memory_entries.financial_fact | memory_entries | "上月批了设备采购30万" | 永久，较快衰减 |
| memory_entries.decision | memory_entries | "决定让王磊试用AI报价" | 永久，中速衰减 |
| raw_weekly_report | weekly_reports.raw_text | 原始周报全文 | **永久保留，不压缩** |
| raw_conversation | conversation_messages | 原始对话文本 | **永久保留** |

**创建触发**：每轮对话后 GLM-4-Flash 异步提取；周报转发时存储原文。

**关键原则**：L1 是不可变的事实记录。一旦提取，只能被新事实取代（supersedes_id），不能被编辑。衰减仅影响注入时的优先级排序，不影响存储。

### L2 二级材料（单次 chatflow 快照输出）

由**一次 chatflow 运行**产生的结构化输出。是某个时间点的"快照"。

| 成果物类型 | artifacts.type | 产出 chatflow | 特征 |
|-----------|---------------|--------------|------|
| 单次周报分析 | weekly_report_analysis | 诊脉·周报分析流 | 时间点快照：某员工某一周 |
| 体检报告 | health_check | 诊脉·体检流 | 时间点快照 |
| 现金流快照 | cash_flow | 管家·现金流预警流 | 时间点快照 |
| 政策扫描结果 | policy_scan | 管家·政策扫描流 | 时间点快照 |
| 会议分析 | meeting_analysis | 诊脉·开会翻译流 | 一次性产物 |
| 政策申领材料 | policy_application | 管家·政策申领流 | 一次性产物 |
| 老板自评快照 | boss_self_assessment | 诊脉·老板盘点流 | 时间点快照（L3 personal_radar 的输入） |
| 企业盘点快照 | enterprise_assessment_snapshot | 诊脉·企业盘点流 | 时间点快照（L3 enterprise_assets 的输入） |
| 初始员工建档 | employee_initial_profile | 诊脉·员工建档流 | 种子数据（L3 employee_profile 的种子） |

**保留策略：全部版本永久保留**。原因：
- 周报分析 v1, v2, v3... 共同构成趋势线，任何一个都不能丢
- 体检报告 v1 vs v2 的对比是"进步感知"的核心
- 现金流快照构成财务趋势
- **不做内容压缩** — full_content 保持原样，token 控制通过注入时只用 summary 字段实现

### L3 三级材料（累积智慧 — 越积越厚）

由多个 L1 和 L2 材料综合而成。**不是某个时间点的快照，而是随时间积累越来越丰富的认知**。

| L3 材料 | artifacts.type | 输入源 | 重合成触发 |
|---------|---------------|--------|-----------|
| **员工综合画像** | employee_profile | L1: 全部employee_fact + boss_observations; L2: 全部weekly_report_analysis + employee_initial_profile | 新周报分析产生; 累积3+条新employee_fact |
| **老板个人画像** | personal_radar | L1: 全部behavior + preference + decision + emotional_state; L2: 全部boss_self_assessment | 新boss_self_assessment产生; 累积5+条新behavior |
| **企业综合画像** | enterprise_assets | L1: business_fact + financial_fact + identity; L2: health_check + cash_flow + enterprise_assessment_snapshot; L3: employee_profile的summary | 新health_check/cash_flow产生; 季度定时 |
| **一页纸战略** | strategy_card | L1: decision + business_fact; L2: health_check + enterprise_assessment_snapshot; L3: enterprise_assets + personal_radar | 老板主动; 累积3+条新decision; 季度提醒 |
| **AI协同空间评估** | ai_automation_analysis | L2: 该员工全部weekly_report_analysis; L1: 该员工的employee_fact | 每次新周报分析后级联重算 |

#### 核心：员工画像的增量更新

**不需要每次读全部20+周的周报**。因为上一版 employee_profile 已经吸收了之前所有历史：

```
重合成输入：
  1. 上一版 employee_profile.full_content (~1000 token) ← 已包含累积智慧
  2. 新增的 weekly_report_analysis (~500 token)  ← 本次新增的L2
  3. 新增的 employee_fact (~300 token)            ← 本次新增的L1
  总输入：~1800 token，远在可控范围内
```

这就像 Git — 不需要从第一个 commit 开始 replay，只需要在 HEAD 上 apply 新的 diff。

#### 12+周数据的 Token 管理：分层摘要策略

当注入 L3 employee_profile 到其他 chatflow 时：
- **最近 4 周**：注入每周的 summary（每周~200 token = 800 token）
- **5-12 周前**：按月合并为 monthly_digest（每月~300 token）
- **12 周以前**：合并为 historical_digest（~500 token）
- **总计**：约 1900-2200 token，可控

monthly_digest 在 Auto Dream（每周日）中生成。historical_digest 在月度摘要超过 3 个月时合并。

### L4 四级材料（跨模块洞察）

| L4 洞察 | 输入源 | 重算触发 |
|---------|--------|---------|
| 撒合就绪度 (sahe_readiness) | L3: team_ai_potential + enterprise_assets; L1: behavior + financial_fact | health_check产生; 季度 |
| 核心卡点 (core_bottleneck) | L1: behavior + emotional_state; L3: personal_radar | boss_self_assessment产生; 照镜干预后 |
| 团队AI潜力 (team_ai_potential) | L3: 所有员工的 ai_automation_analysis | 任何员工的ai_automation更新 |
| 老板成长路径 (boss_growth_path) | L3: personal_radar; L1: decision + behavior | personal_radar更新 |
| 当前最优先行动 (priority_action) | L4: core_bottleneck + team_ai_potential; L3: strategy_card; L2: 最新周报分析 | 每次周报/体检/照镜后 |
| 跨模块信号 (cross_module_signal) | 全部 L2 和 L3 | Auto Dream 每周检测 |

**保留策略**：保留最近 5 个版本（含 evidence 来源引用），用于趋势分析。

## 二、17 个 Chatflow 的上下文配方表

### 设计原则
1. 每个 chatflow 只注入它需要的材料，不注入无关数据
2. Token 预算：总注入不超过 4K（留 4K 给对话历史，总共 8K）
3. L3/L4 用 summary 注入，不用 full_content
4. L2 历史版本有明确数量上限
5. 注入格式：结构化自然语言（Markdown 风格，非 JSON）

## 三、周报→员工画像 Pipeline（核心数据管线）

### 流程图

```
老板转发周报
    │
    ▼
存储 raw_text → weekly_reports (L1, 永久)
    │
    ▼
诊脉·周报分析 chatflow（注入 L1+L2+L3 上下文）
    │
    ▼
产出 weekly_report_analysis (L2)
    │
    ├─→ 提取 employee_fact → memory_entries (L1)
    ├─→ 重算 ai_automation_analysis (L3)
    ├─→ 重合成 employee_profile (L3) ←── 增量更新：旧画像 + 新增信息
    ├─→ 重算 team_ai_potential (L4)
    ├─→ 重算 priority_action (L4)
    └─→ 更新 familiarity_score
```

### 重合成细节

**输入**：
1. 上一版 employee_profile.full_content (~1000 token)
2. 新增的 weekly_report_analysis.full_content (~500 token)
3. 新增的 employee_fact (~300 token)
4. boss_observations 更新 (~200 token)

**输出**：新版 employee_profile，version+1

**GLM-5 调用**：总输入 ~2000 token + system prompt ~500 token = 2500 token
- 成本：$0.002
- 延迟：< 10s

### 边缘情况

| 场景 | 处理方式 |
|------|---------|
| 新员工只有1份周报 | profile ≈ initial_profile + 第一次周报分析；标注"(数据量少)" |
| 老板3个月没提某员工 | Auto Dream标记stale；下次提及时一树提示"了解可能不太准了" |
| 跨员工相似模式 | Auto Dream每周做跨员工key_findings模式识别 → cross_module_signal |
| 全量重建需求 | 读取全部 weekly_report_analysis，用分层摘要策略控制token |

## 四、数据保留策略汇总

| 层级 | 数据 | 保留 | 原因 |
|------|------|------|------|
| L1 | conversation_messages | 永久 | 审计/回溯 |

| L2 | policy_scan | 最近3版，更早仅保留summary | 历史匹配价值有限 |

| L3 | employee_profile 等 | **最近3版 full_content**，更早仅保留summary | 控制存储，summary永久 |

## 六、Auto Dream 周期任务

每周日凌晨运行：

1. **时间归一化**：memory_entries 相对时间 → 绝对时间
2. **矛盾清理**：删除 is_active=false 且超过30天的记忆
3. **合并同类项**：GLM-4-Flash 判断同员工相似 employee_fact 并合并
4. **月度摘要生成**：检查上月 weekly_report_analysis 是否需合并为 monthly_digest
5. **L3 版本清理**：超过3个版本的删除最早版本 full_content（保留 summary）
6. **L3 批量重合成**：对所有 is_stale=true 的 L3 材料执行重合成
7. **L4 全量重算**：所有 user_insights 重新计算
8. **跨员工模式检测**：读取本周全员 key_findings → GLM-5 模式识别 → cross_module_signal
9. **了解度更新**：重算 users.familiarity_score
