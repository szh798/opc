# 创业者指导小程序产品需求与软件架构方案

基于当前小程序前端页面、现有后端接口和产品文档，对“为创业者提供指导和建议的小程序”进行产品需求梳理、产品设计和软件架构设计。

## 1. 基于现有前端页面的产品判断

### 1.1 现有页面已经表达出的产品骨架

当前页面结构不是一个“工具集合”，而是一个以对话为中枢的创业陪跑系统：

- `welcome`：承接品牌认知和首次转化
- `conversation`：真正的主操作系统，负责诊断、引导、执行、反馈
- `ai-assistant` / `ip-assistant`：本质上是对话场景切换，而不是独立产品
- `project-detail`：项目级工作台，沉淀项目对话和成果
- `profile`：用户级长期资产档案
- `tree`：成长过程可视化，强化“我在变强”的感知
- `weekly-report` / `monthly-check` / `social-proof` / `milestone`：留存、复盘、召回机制
- `share-preview`：结果外显和传播

### 1.2 当前产品的真实定位

从页面和交互设计看，这个产品最适合的定位不是“AI 问答助手”，而是：

**一个面向创业者的一人公司 AI 陪跑操作系统。**

它解决的不是“用户有没有问题”，而是：

- 用户不知道该做什么
- 用户知道但做不动
- 用户做了但没有节奏
- 用户有结果但没有沉淀
- 用户有业务但不会系统化和放大

### 1.3 当前前端已经验证的核心产品假设

现有页面已经证明了 5 个关键假设：

1. 主入口应该是对话，不是复杂表单。
2. 用户需要的不只是建议，还需要“下一步动作”。
3. 创业建议必须沉淀为结构化成果，否则无法复用。
4. 创业产品需要强留存机制，不能只靠用户主动提问。
5. 产品价值要从“聊天体验”升级为“创业进展可见”。

## 2. 用户需求梳理

## 2.1 目标用户分层

结合现有页面和产品文档，可以把核心用户归纳为五类：

### A. 迷茫探索者

特征：

- 想创业
- 有焦虑
- 没方向
- 不知道自己能卖什么

核心需求：

- 帮我发现我能做什么
- 帮我筛掉不适合的方向
- 给我一个低风险起步路径

### B. 行动瘫痪者

特征：

- 有想法
- 长期卡住
- 信息很多但没有行动

核心需求：

- 帮我把想法拆成第一步
- 帮我克服拖延和恐惧
- 给我一个能在 15-30 分钟内完成的动作

### C. 单打独斗者

特征：

- 已经接单或有副业
- 收入不稳定
- 做法不标准

核心需求：

- 帮我稳定获客
- 帮我整理产品和报价
- 帮我把经验变成可复制的 SOP

### D. 放大规模者

特征：

- 已经有收入
- 时间被交付占满
- 想用 AI 和系统来放大

核心需求：

- 帮我找到可以 AI 化的环节
- 帮我搭自动化工作流
- 帮我把个人能力变成规模能力

### E. 合规求助者

特征：

- 业务开始跑起来
- 开始面临公司、财税、薪资、园区政策等现实问题

核心需求：

- 帮我看懂合规事项
- 帮我做关键时间节点提醒
- 帮我把公司经营动作结构化

## 2.2 用户的共性需求

无论用户处在哪个阶段，本质上都有 7 个共同需求：

### 1. 方向判断需求

用户需要知道：

- 我适合做什么
- 哪个方向更有胜率
- 哪个方向更适合现在的自己

### 2. 决策降噪需求

创业者面对的信息太多，最缺的不是内容，而是：

- 过滤
- 判断
- 优先级

### 3. 降低行动门槛需求

用户需要的不是“知道更多”，而是：

- 今天先做哪一件事
- 这件事怎么开口
- 做完之后下一步是什么

### 4. 外部陪跑需求

创业的真正难点之一是长期独自推进，因此用户需要：

- 有人持续提醒
- 有人及时反馈
- 有人追问结果

### 5. 成果沉淀需求

用户不希望每次都从头开始，所以需要：

- 资产画像
- 商业方向
- 产品结构
- 定价策略
- 触达话术
- 周报月报

这些都能长期积累。

### 6. 进展可视化需求

创业很容易陷入“我好像没进步”的感受，所以产品必须让用户看到：

- 我完成了什么
- 我现在处在哪个阶段
- 我离下一阶段还有多远

### 7. 经营系统化需求

当用户进入更成熟阶段，会从“我要赚钱”升级为：

- 我要更稳定
- 我要更省力
- 我要更可复制
- 我要更合规

## 2.3 用户需求的优先级排序

按创业者真实使用频率和价值密度，建议优先级如下：

### P0

- 创业方向诊断
- 对话式建议
- 今日任务
- 项目沉淀
- 成长反馈

### P1

- 产品化与定价建议
- 周报/月报
- 分享传播
- IP 助手 / AI 助手

### P2

- 公司经营面板
- 财税与园区辅助
- 自动化流程搭建
- 更强的召回与增长体系

## 3. 从用户需求倒推产品设计

## 3.1 产品定位

### 产品定义

这是一个帮助创业者从“迷茫”走到“行动”，从“行动”走到“拿结果”，再从“拿结果”走到“系统化经营”的 AI 陪跑小程序。

### 产品价值主张

- 帮你找到方向
- 帮你做出第一步
- 帮你持续推进
- 帮你沉淀成果
- 帮你看到成长

## 3.2 产品核心目标

建议把产品目标定义成三层：

### 目标 1：缩短创业启动时间

让用户更快从“想创业”进入“开始验证”。

### 目标 2：提升连续行动率

让用户不是只聊一次，而是每天都能推进一点。

### 目标 3：形成创业资产库

让对话结果转化为可持续复用的项目资产、个人资产和经营资产。

## 3.3 核心产品策略

### 策略一：对话是入口，结构化是价值

用户进入产品时，应该通过对话感到轻松；离开产品时，应该拿到结构化成果。

### 策略二：建议必须落到动作

每一轮有效对话，都应该尽量输出以下三类之一：

- 一个判断
- 一个动作
- 一张成果卡

### 策略三：以项目为经营单元

创业不是抽象聊天，必须绑定在项目上推进。因此项目应该成为主业务对象。

### 策略四：以成长反馈驱动留存

留存不靠“内容更新”，而靠“你在成长”的感知。

### 策略五：从陪聊升级为陪跑

产品不能停留在给建议，要进入：

- 任务
- 反馈
- 追问
- 复盘
- 成长

的闭环。

## 3.4 产品信息架构

建议的信息架构如下：

### 1. 获客层

- 欢迎页
- 品牌价值表达
- 社会证明
- 登录引导

### 2. 核心交互层

- 主对话页
- 多角色切换
- 快捷回复
- 流式回复
- 输入框和项目入口

### 3. 工作台层

- 项目列表
- 项目详情
- 项目成果
- 公司面板

### 4. 用户资产层

- 我的档案
- 成长树
- 当前阶段
- 长期优势画像

### 5. 反馈与留存层

- 每日任务
- 里程碑
- 周报
- 月度体检
- 社会证明召回

### 6. 传播层

- 分享海报
- 分享文案
- 成果外显

## 3.5 关键用户旅程设计

### 旅程一：首次进入用户

1. 用户进入欢迎页
2. 进入对话，完成微信登录
3. 一树识别用户所处状态
4. 系统进入对应路径
5. 产出第一个明确建议或资产卡
6. 自动生成今日任务
7. 用户完成第一次留存闭环

产品目标：

- 让用户在第一次使用的 5 分钟内感受到“被理解”和“能开始”

### 旅程二：有方向但推进困难的用户

1. 进入对话
2. 描述当前卡点
3. AI 拆解问题
4. 生成可执行动作
5. 用户勾选任务
6. 系统追问结果
7. 基于反馈继续给下一步

产品目标：

- 从内容陪伴升级成执行陪跑

### 旅程三：已有业务的用户

1. 进入项目详情
2. 围绕项目持续对话
3. 产出成果卡
4. 进入产品化 / 定价 / 获客 / 复购建议
5. 在周报月报里看到经营变化

产品目标：

- 从单点建议升级为经营管理辅助

## 4. 软件架构设计

## 4.1 架构设计原则

### 原则 1：先做模块化单体，不急着拆微服务

当前产品仍处于高迭代阶段，最适合的是：

**模块化单体 + 清晰领域边界 + 可演进事件机制**

原因：

- 产品逻辑还在快速变化
- 团队协作成本需要低
- 小程序后端请求链路需要简单稳定
- 很多能力都围绕同一个用户、项目和对话上下文

### 原则 2：对话是业务中台，不是独立边角功能

聊天系统必须和：

- 用户画像
- 项目状态
- 任务系统
- 成长系统
- 报告系统

深度联动。

### 原则 3：结构化数据优先于纯文本

AI 输出最终要尽量落成：

- 状态
- 卡片
- 任务
- 报告
- 快捷回复

而不是只存一段文字。

### 原则 4：业务域和 AI 编排解耦

AI 负责“推理与生成”，业务系统负责：

- 权限
- 状态
- 数据存储
- 生命周期
- 规则约束

## 4.2 推荐的总体架构

### 逻辑分层

```text
微信小程序前端
  -> BFF/API 层（NestJS）
    -> 业务域服务层
      -> AI 编排层
      -> 规则引擎 / 任务引擎 / 成长引擎
        -> PostgreSQL / Redis / 对象存储
          -> 微信登录 / LLM(Dify) / 消息与分享能力
```

### 说明

- 前端负责页面状态、交互体验、轻量本地状态
- BFF 负责接口聚合、鉴权、场景收口
- 业务域服务负责核心业务规则
- AI 编排层负责意图识别、角色路由、上下文组装、结构化输出
- 数据层负责长期记忆、项目资产、快照和分享文件

## 4.6 关键业务流设计

### 流程一：登录与启动

```text
用户进入 welcome
  -> 进入 conversation
  -> 微信登录
  -> /bootstrap 拉取用户、项目、最近对话、工具入口
  -> 渲染初始场景
```

### 流程二：对话与 AI 回复

```text
用户发消息
  -> Conversation Service 校验场景与上下文
  -> Agent Orchestrator 选择角色和 prompt
  -> Context Builder 注入用户、项目、任务、最近成果
  -> LLM/Dify 返回文本或结构化输出
  -> 系统写入 Message / Artifact / Task 等
  -> 前端轮询 stream event 展示流式回复
```

### 流程三：任务闭环

```text
系统下发任务
  -> 用户勾选完成
  -> Task Service 记录完成状态
  -> Feedback Service 生成追问
  -> 用户补充结果
  -> AI 给出下一步建议
  -> Growth Engine 更新成长快照
```

### 流程四：项目成果沉淀

```text
项目对话产生有效内容
  -> Artifact Builder 识别成果类型
  -> 写入 ProjectArtifact
  -> 项目结果页展示
  -> 用户可分享或继续追问
```

### 流程五：成长与报告

```text
消息 / 任务 / 项目 / 反馈变化
  -> Growth Engine 重新计算阶段
  -> Report Engine 生成周报 / 月报 / 召回内容
  -> 前端通过页面或对话卡片展示
```

## 4.7 AI 编排层设计

这是产品最重要的技术模块，建议单独抽象。

### AI 编排层应包含的能力

#### 1. Intent Router

判断用户当前属于哪一类需求：

- 探索方向
- 推进行动
- 项目经营
- 情绪卡点
- 公司事务

#### 2. Role Resolver

根据意图切换角色：

- 一树
- 挖宝
- 搞钱
- 扎心
- 管家

#### 3. Context Builder

在发给模型前，拼装：

- 用户画像
- 当前阶段
- 当前项目
- 最近成果
- 今日任务
- 最近反馈

#### 4. Structured Output Parser

将模型输出解析为：

- 文本消息
- 快捷回复
- 任务卡
- 成果卡
- 状态更新

#### 5. Memory Writer

把有价值的信息写回：

- User Profile
- Project Artifact
- Task Feedback
- Growth Snapshot

#### 6. Guardrail & Compliance

控制高风险内容：

- 财税建议提示免责
- 避免夸大承诺
- 避免违规商业建议

## 4.8 数据与基础设施建议

### 当前合适的基础设施

- PostgreSQL：主业务数据
- 本地文件存储 / 对象存储：海报与分享资源
- Dify：LLM 编排入口

### 建议补充的基础设施

#### Redis

用途：

- 会话短期状态
- 流式回复缓存
- 幂等 key
- 频控

#### 消息队列

用途：

- 周报月报异步生成
- 分享海报异步生成
- 成长快照重算
- 推送任务和召回消息

在 MVP 阶段可以先用数据库 + 定时任务，等量上来再接入。

#### 对象存储

建议未来把海报文件从本地存储迁移到对象存储，便于：

- 扩容
- CDN 分发
- 更稳定的分享访问

## 4.9 非功能性设计

### 安全

- 登录态校验
- Token 轮换
- 数据隔离到 userId
- 敏感操作审计日志

### 稳定性

- AI 超时容错
- Dify 不可用降级
- 分享生成失败回退
- 关键链路可重试

### 可观测性

建议增加：

- 请求日志
- 对话成功率
- AI 超时率
- 登录成功率
- 任务完成率
- 周留存 / 月留存
- 从 onboarding 到项目创建的转化率

### 性能

- `/bootstrap` 聚合返回
- 对话流使用事件流或轮询事件表
- 报告和海报异步化
- 快照类数据避免每次全量重算

## 4.10 推荐的演进路线

### 第一阶段：把当前能力做成闭环

重点：

- 主对话链路稳定
- 项目和成果稳定沉淀
- 任务闭环可用
- 成长树和周报月报可信

### 第二阶段：做强 AI 编排和结构化输出

重点：

- 更准确的用户阶段识别
- 更稳定的成果卡生成
- 更清晰的角色切换
- 更有效的快捷回复和行动建议

### 第三阶段：做深经营能力

重点：

- 经营看板
- 财税和公司事项流程化
- 自动化工作流
- 更多真实商业动作接入

### 第四阶段：做增长和平台化

重点：

- 分享裂变
- 用户分层运营
- 行为分析和个性化推荐
- 从陪跑工具演进为创业操作系统

## 5. 对当前项目的结论与建议

## 5.1 当前产品方向是对的

当前页面结构和后端接口已经说明，这个产品不是简单的创业资讯产品，而是：

**以对话为入口、以项目为单位、以成果为资产、以成长为留存的创业陪跑系统。**

这个方向是成立的，而且比“纯 AI 聊天”更有产品壁垒。

## 5.2 当前最应该做的不是加更多页面，而是强化闭环

建议近期优先做强这四件事：

1. 首次 onboarding 到项目创建的闭环
2. 对话到成果卡沉淀的闭环
3. 今日任务到反馈追问的闭环
4. 成长树 / 周报 / 分享的反馈闭环

## 5.3 架构上最适合的策略

建议继续走：

**小程序前端 + NestJS 模块化单体 + PostgreSQL + AI 编排层**

而不是过早拆成微服务。

## 5.4 后续最关键的技术投入点

如果只能重点投入三个方向，我建议是：

1. AI 编排层与结构化输出
2. 项目 / 成果 / 任务 / 成长的数据闭环
3. 报告、分享、召回带来的留存和增长机制
