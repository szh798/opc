## 三、本周重点工作

### 1. 挖宝商业方向 V1 和项目承接

本周明确了挖宝智能体的 V1 范围：每个用户最多 1 个隐藏 opportunity draft 和 1 个 active project，主链路固定为“3 个候选方向 -> 选择方向 -> 深聊持久化 -> 立项摘要 -> 确认立项 -> 3 天一轮跟进”。

对应落地包括项目数据模型、Dify opportunity flow、项目入口修复、项目详情和周期任务承接。重点修复了“点击打开项目跳到一树正在开发，而不是进入项目模块”的问题，并梳理了项目打开、路由流处理和上下文恢复。

### 2. 资产报告流式进度反馈

本周完成资产报告从普通生成体验到真实流式进度体验的升级。后端以 SSE 事件输出 `card.created`、`card.patch`、`card.completed`、`final_report.created`、`stream.done` 等事件，前端通过 `asset_report_progress` 卡片实时渲染步骤、进度条、资产标签和雷达预览。

关键技术点包括：

- 统一卡片协议：后端事件里的 `card_type` 固定为 `asset_report_progress`。
- 后端新增真实流式接口和断线恢复事件读取。
- 增加流式 XML / markup 屏蔽逻辑，避免 Dify 的内部标签分块露出到前端。
- 前端支持进度卡 create-on-patch、最终报告卡去重、进度卡和最终报告卡生命周期切换。
- 修复资产报告生成过程中出现两个报告卡、进度卡不显示、最终卡片提前出现等问题。

### 4. 每日任务 / 本轮推进

本周继续打磨任务推进体验，包含“今日任务”改为 `Suggestion`、3 个任务完成后仍未推进的问题排查，以及本轮推进状态和项目 `nextValidationAction` 的关系梳理。

4 月 29 日日报中已记录：页面上的“本轮目标”是静态轮次目标，真正下一步状态来自 `project.nextValidationAction`；后端需要在任务完成后推进跟进轮次，并同步到前端。

### 5. 登录、手机连接与真机巡检

本周完成了 Aliyun SMS 登录相关提交，并继续排查微信登录、手机号登录和真机访问问题。

手机连接方面完成：

- 安装并配置 Android Platform Tools。
- 连接 Android 真机并通过 ADB 巡检。
- 尝试安装 / 使用 scrcpy。
- 生成 13 张 ADB 巡检截图，覆盖欢迎页、CTA、菜单、登录态、输入框锁定和 pending 状态。
- 发现手机无法访问电脑局域网后端的根因：电脑 WLAN 网络为 Public，3000 端口防火墙规则只放行 Private。
- 在无管理员权限修改防火墙的情况下，使用 `adb reverse tcp:3000 tcp:3000` 建立 USB 反向代理。
- 手机端访问 `http://127.0.0.1:3000/ready` 成功返回 200。

`/ready` 真机链路数据：`db_ping` 约 2ms，`db_write` 约 3ms，`dify_reachable` 约 935ms，总耗时约 940ms。

### 6. Lunel 手机连接 Codex

本周研究并接入 Lunel，让手机端能够连接电脑上的 VS Code/Codex 工作区。已确认当前连接不是必须同一局域网，Lunel Agent 连接走公网网关；只有手机访问电脑本地后端时才需要局域网、端口转发或公网 HTTPS。

## 六、当前风险与阻塞

1. 生产 HTTPS 仍是最大阻塞。`api.atreeagent.com/health` 和 `/ready` 必须在公网 HTTPS 下返回 200，才能进入完整上线 smoke。
2. 曾进入 git 历史的 smoke token 必须轮换，不能带旧 token 上线。
3. 微信正式授权链路仍需要在真机和正式 appid 下做最终确认。
4. Windows 防火墙 Public 网络阻止手机直连电脑 3000 端口，当前 Android 可通过 USB reverse 绕过，但这不是生产问题，只影响本地真机调试。
5. 资产报告进度卡涉及 SSE 顺序、前端渲染和最终卡片去重，已做修复，但仍建议在开发者工具和真机上用同一条真实资产盘点流再做一次端到端验收。
