# 一树 OPC 本地部署可行性评估

## 一、你想自部署的 `glm-4-air` 和 `glm-4-flash` —— 其实下载不到

智谱的模型分两类,名字容易搞混:

| 类别 | 例子 | 能不能下载? |
|---|---|---|
| **API 商用模型**(我们现在在用的) | `glm-4-air`、`glm-4-flash`、`glm-4-plus` | **不能**。闭源,只能调 API。你付的钱其实就是在买这个 |
| **开源权重模型**(可以拖到本地跑) | `GLM-4-9B-Chat`、`ChatGLM3-6B` | 能。Hugging Face 上能下,用 vLLM / ollama 能跑 |

也就是说:**你付给智谱的那几分钱,买的是 "air/flash" 这个服务本身**,不是买模型文件。本地 5090 上你永远跑不起来"同一个 glm-4-air"。

但是 —— **你真正想做的事是能做的**,就是"用一个差不多等级的开源模型,部署在家里的 5090 上,替代掉这两个 API 调用"。下面我按这个真实需求继续聊。

## 二、5090 上能跑什么(对应我们现在的两个任务)

5090 是 32GB 显存(GDDR7),算是消费卡的天花板。按我们现在的两个任务选合适的开源模型:

| 任务 | 当前 API | 推荐的本地替代 | 占显存(4bit量化) |
|---|---|---|---|
| L1 事实抽取(高频) | glm-4-flash | **Qwen2.5-7B-Instruct** 或 **GLM-4-9B-Chat** | 5-7GB |
| L2 对话摘要(低频) | glm-4-air | **Qwen2.5-14B-Instruct**(如果要质量就上 Qwen2.5-32B 4bit) | 9GB / 18GB |

一张 5090 同时跑 7B + 14B 没问题,还剩十几 GB 给上下文。用 vLLM 做推理服务,能自动批处理(把多个用户的请求塞到一起算,显著提高吞吐)。

## 三、能撑几个用户?—— 粗算一笔

先说结论,中间的算法你可以跳过。

**能稳定服务的规模(5090 单卡,两个小模型并存):**

| 口径 | 大概值 |
|---|---|
| **同时在打字** 的用户(真正并发请求) | **8~15 人** |
| **同一时段活跃**(5 分钟内有消息往来) | **30~60 人** |
| **日活用户 DAU** | **300~800 人**(假设用户平均每天活跃 10 分钟) |

**算法(你不想看可跳):**

假设 Qwen2.5-14B 在 5090 上的聚合吞吐约 150~200 token/秒(开 vLLM batching)。
- 一次 L1 抽取 ≈ 200 输出 token ≈ 1~1.5 秒 GPU 时间
- 一次 L2 摘要 ≈ 300 输出 token ≈ 2 秒 GPU 时间(但频率只有 L1 的 1/20)
- 一个活跃用户每 30~60 秒发一条消息 → 每个活跃用户约占 0.02~0.05 的 GPU
- 单 5090 总预算 = 1 个 "GPU·秒 / 秒"
- 理论上限 = 20~50 个活跃用户;扣掉空跑、突发、L2 尖峰,**安全值 30 左右**

**关键警告**:这是"记忆任务"的量。如果你以后**还想用本地模型直接生成主对话回答**(替代 Dify 那边的 chat 主模型),那需求量是 5~10 倍,5090 立刻扛不住,得看 80G 卡或多卡了。

## 四、"云服务跑后端 + 本地跑小模型" —— 能做,但有坑

架构长这样:

```
用户手机
   ↓ HTTPS
云服务器(阿里云/腾讯云)
  ├─ Nest 后端
  ├─ PostgreSQL
  └─ 调 Dify 主对话(还是走 Dify 云)
         ↓
         调"记忆服务"(你家的 5090)
         ↓ 走 frp / Cloudflare Tunnel 内网穿透
         家里的 5090 +  vLLM + Qwen2.5
```

**技术上完全 OK,但你作为新手要先过这 5 个坑:**

**坑 1:家里没有公网 IP**
家宽 99% 是运营商 NAT,云服务器没法直接连到你家。三种解决办法:
- **Cloudflare Tunnel**(推荐给新手):免费,5 分钟装好,全程 HTTPS,不暴露家里 IP
- **frp / nps**:自己搭,需要一台便宜的公网 VPS 做中转
- **Tailscale**:组 VPN,云服务器和家 PC 拉到同一个内网里

**坑 2:家宽上行带宽小**
很多家宽下行 500M、上行只有 30M。L1/L2 都是短文本,上行占用不多,**这个不是主要问题**。

**坑 3:家里断电 / 断网 / 跳闸 / 误关机**
这是最要命的问题。云服务器是 4 个 9 可用性(一年宕机 < 1 小时),家里能做到 2 个 9 就不错了(一年宕机几十小时)。宕机时你的用户**感觉不到记忆被吞**,因为 fire-and-forget 不会让他们看到错误,但数据会静悄悄地丢掉 —— 这比直接报错更糟。

**坑 4:电费 + 噪音 + 发热**
5090 7x24 跑中载,大约 200~350W。一个月电费 100~200 块人民币(北上广商用电更贵)。还有风扇噪音和夏天室温问题 —— 夏天会很难受。

**坑 5:维护成本**
家里的 GPU 服务器不是装完就不管了:
- vLLM 升级、CUDA 驱动、Qwen 新版,得定期维护
- 显卡挂了、风扇坏了你得自己排查
- Windows 更新可能强制重启 → 服务下线

这些时间成本在云上是运营商帮你承担的。

## 五、最关键的点 —— 算一下到底划不划算

把账摊开:

**现状(API 方案)**
- 当前用户量假设 10~50 DAU
- L1 每天调用:50 用户 × 每人 20 轮 × 每轮 1 次 = 1000 次/天,平均 300 token
- L2 每天调用:50 用户 × 每人 2 次切换 × 每次 1500 token in + 300 token out
- **Zhipu 月成本 ≈ 10~50 块人民币**(flash 超便宜,air 贵但量小)

**自部署方案**
- 5090 硬件(你已经有就算沉没成本;没有要 ¥12000~15000)
- 电费:¥150/月
- Cloudflare Tunnel:免费
- 维护时间:平均每周 1~2 小时(折算 ¥200/月人力成本,你自己做就当零)
- **月运营 ≥ ¥150**

**结论:**
- 在 **DAU < 500** 这个段位,API 比自部署便宜得多,而且省心 10 倍
- 自部署的盈亏平衡点大约在 **DAU 1000 以上**,或者你有隐私合规需求(用户数据不能出家门)必须自部署
- 到 **DAU 5000+**,API 月成本可能上千,这时自部署才真正开始赚钱

## 六、给你这个项目的具体建议(新手友好版)

我会这样做,你可以参考:

1. **现阶段不要动**。产品还没几个真实用户,月 API 账单不到一顿火锅钱。把 5090 的精力留着
2. **先把 API 跑熟,收集数据**。等 DAU 过 200,我们再讨论是否自部署
3. **如果你就是想折腾 / 学习本地部署**,最低成本方案是这样:
   - 在 5090 上用 `ollama` 或 `vLLM` 起一个 Qwen2.5-7B
   - **不替换** 现在的 API 调用,而是在 [chatflow-summary.service.ts](backend/src/memory/chatflow-summary.service.ts) 里加一个**双写模式** —— 同一段对话既调 API 也调本地模型,把两边结果都记下来
   - 跑两周,对比一下哪边的摘要质量更好、延迟各是多少
   - 有了真实对比数据再决定要不要切换
4. **不要让主对话走本地**。主对话是用户**直接看到的**,延迟要求高、质量要求高、还得支持 stream。本地小模型在这块会翻车。主对话继续走 Dify / Zhipu 云 API
5. **真要上云+家混合架构**,先用 Cloudflare Tunnel 做内网穿透,别自己折腾 frp(新手容易踩坑)

## 七、一句话总结

**技术上完全可行,经济上现阶段不划算,稳定性上家里扛不住生产流量**。等有真实用户规模了再上;现在想练手就"双写对比",别直接切。
